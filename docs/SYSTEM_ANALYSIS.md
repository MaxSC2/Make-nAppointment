# Системный анализ и архитектура хранения данных (MedPlatform)

> **Назначение:** Анализ соответствия проекта целевому видению врача-радиолога, проектирование архитектуры двухуровневого хранения снимков (Hot/Cold Storage Tiering) и Gap-анализ текущих отклонений реализации от клинических целей платформы.

---

## 1. Концептуальное видение: Пациенто-центричная экосистема врача

Целевая платформа — это не просто разрозненные модули электронной очереди и PACS-вьювера, а **интегрированная медицинская рабочая среда (EHR / PACS Worklist)**, спроектированная вокруг рабочего процесса врача-радиолога.

### Ключевые принципы целевого состояния:
1. **Принцип неснижаемого накопления (Lossless EHR):** Вся история обращений, статус в очереди, результаты исследований (DICOM) и подписанные протоколы навечно аккумулируются в персональной карте пациента. Удаление данных запрещено; допустим только перевод в архив.
2. **Пациенто-центричный контекст (Patient-Centric Folders):** Врач оперирует не "плоским списком исследований из PACS", а медицинскими картами пациентов. Открывая карту, он мгновенно видит полную хронологию:
   * История посещений и текущее положение в очереди.
   * Список всех прошлых исследований с быстрым доступом.
   * Предыдущие протоколы для оценки динамики (сравнение "было / стало").
3. **Скорость и непрерывность визуализации:** Врач должен за 1 клик и менее чем за 1 секунду увидеть, что за снимок привязан к пациенту. Медленная загрузка тяжелых серий (особенно КТ/МРТ) блокирует клинический процесс.

## 2. Архитектура двухуровневого хранения снимков (Hot/Cold Storage Tiering)

DICOM-данные накапливаются лавинообразно. Одно исследование КТ/МРТ может весить от 100 МБ до 1.5 ГБ. Со временем данные «охлаждаются»: частота обращений к снимкам старше 30 дней падает на 95%. 
Для предотвращения переполнения быстрых SSD-дисков и снижения затрат на инфраструктуру проектируется архитектура двухуровневого хранения (Tiering).

### Схема архитектуры хранения снимков

```
                     [ Врач в браузере (React SPA) ]
                                    │
                                    ▼
                      [ RIS / PACS-фасад (FastAPI) ]
                                    │
                  ┌─────────────────┴─────────────────┐
                  ▼ (WADO-RS / DICOM-web)             ▼ (Фоновый тиринг)
         ┌──────────────────┐               ┌────────────────────┐
         │ HOT STORAGE TIER │               │ COLD STORAGE TIER  │
         │ (Быстрый доступ) │               │ (Архивное облако)  │
         └────────┬─────────┘               └─────────┬──────────┘
                  │                                   │
                  ▼                                   ▼
          ┌──────────────┐                     ┌──────────────┐
          │   Orthanc    │                     │   S3 Cloud   │
          │ (SQLite/PG)  │                     │  or NAS-HDD  │
          │   на SSD     │                     │ (Compressed) │
          └──────────────┘                     └──────────────┘
```

### Спецификация уровней хранения

| Характеристика | Горячее хранилище (Hot Tier) | Холодное хранилище (Cold Tier) |
|---|---|---|
| **Назначение** | Оперативная работа, активные исследования, свежие снимки. | Долговременный архив, исторические снимки для динамики. |
| **Физический носитель**| Локальные SSD высокой скорости (NVMe). | Дешевые сетевые диски (NAS), медленные HDD (RAID) или S3 (Glacier / IA). |
| **Критерий размещения**| Новые снимки (<30 дней) ИЛИ снимки активных пациентов в очереди. | Старые снимки (>30 дней) И снимки пациентов без активных заказов. |
| **Формат хранения** | Сырой DICOM в Orthanc без сжатия (для мгновенной WADO-RS передачи). | Сжатые DICOM (lossless JPEG-LS / JPEG 2000) или архивированные ZIP в S3. |
| **Время доступа (TTFB)**| **< 100 ms** | **1.5 – 3.0 сек** (требуется автоматический разогрев / декомпрессия). |

### Механизм автоматического тиринга (Data Lifecycle Management)

Тиринг управляется фоновым демоном (Celery-воркером / System Daemon в RIS) по следующим правилам:

#### 1. Правило деградации (Заморозка / Freezing):
Фоновый скрипт запускается ежесуточно в 03:00 и выполняет:
```sql
SELECT study_uid FROM studies 
WHERE created_at < NOW() - INTERVAL '30 days' 
  AND status = 'completed'
  AND study_uid NOT IN (SELECT study_uid FROM active_queue_studies);
```
Для каждого найденного исследования:
1. Извлекается оригинальный DICOM-файл из Orthanc Hot-хранилища.
2. Сжимается с помощью `pydicom` / `gdcmconv` в формат **Lossless JPEG-LS** (сжатие ~3:1 без потери диагностических качеств).
3. Загружается в Cold Storage (NAS или S3 Bucket `/archive/{patient_id}/{study_uid}.zip`).
4. В базе данных RIS статус исследования меняется на `cold`, локальный файл в Orthanc удаляется с помощью `DELETE /studies/{id}` для освобождения места. В PACS-фасаде остаётся легковесная запись-заглушка (метаданные).

#### 2. Правило регидратации (Отогрев / Rehydration / Recall):
Когда врач открывает карту пациента со старым снимком (статус `cold`) или нажимает «Открыть в просмотрщике»:
1. PACS-фасад перехватывает запрос и видит статус `cold`.
2. Система отправляет врачу сигнал в интерфейс: *"Снимок восстанавливается из архива..."* (спиннер загрузки).
3. Фоновый процесс скачивает архив из Cold Storage, распаковывает / декомпрессирует DICOM файлы.
4. Снимки фоново импортируются обратно в Orthanc (`POST /instances` через REST API).
5. Статус исследования обновляется на `hot`, обновляется дата последнего обращения `last_accessed_at = NOW()`.
6. Снимок мгновенно открывается во вьювере. Через 7 дней неактивности он снова автоматически "заморозится".

## 4. Дорожная карта выравнивания проекта (Action Plan)

Чтобы максимально приблизить текущий проект к медицинской цели и разгрузить хранилище, рекомендуется выполнить следующие шаги:

### Шаг 1: Оптимизация производительности (Quick Wins)
* **Цель:** Снять bottleneck скорости загрузки снимков.
* **Что сделать:** 
  1. Отредактировать `vite.config.ts`, исправить прокси `/dicom` на правильный путь DICOMweb Orthanc.
  2. Перенаправить `DwvViewer.tsx` на прямой забор DICOMweb, разгрузив FastAPI. Скорость вырастет до 20 раз.

### Шаг 2: Внедрение превью для врача
* **Цель:** Дать врачу "быстрый и лёгкий" просмотр снимка пациента без входа во вьювер.
* **Что сделать:**
  1. На странице `DoctorPage.tsx` рядом с талоном активного пациента вывести превью снимка, используя `/api/v1/studies/{uid}/preview` (генерирует PNG на бэкенде через pydicom).
  2. На странице `OrdersPage.tsx` встроить колонку "Миниатюра" со всплывающим окном полного размера при клике.

### Шаг 3: Переход на пациенто-центричную структуру БД
* **Цель:** Связать очередь, заказы и снимки в единую медицинскую карту.
* **Что сделать:**
  1. На бэкенде (в `db/models/ris.py` и `pacs_facade.py`) добавить сущность `Patient` (ФИО, полис, СНИЛС, дата рождения).
  2. Связать все исследования по `Patient.id`.
  3. На фронтенде создать страницу `PatientEhrPage.tsx` — карточка пациента, где собрана вся его история: талоны в очереди, список исследований, файлы протоколов.

### Шаг 4: Разработка модуля хранения (Storage Tiering)
* **Цель:** Изолировать старые снимки, защитить диск от переполнения.
* **Что сделать:**
  1. Реализовать python-скрипт `tiering_daemon.py` в бэкенде.
  2. Добавить статус `is_cold` (boolean) в таблицу `studies`.
  3. При обращении к `/api/v1/instances/{id}/dicom` проверять статус. Если `is_cold=True`, запускать декомпрессию и импорт из архивного NAS обратно в Orthanc (регидратация) перед отдачей файла во вьювер.

## 5. Архитектура интеграции ИИ-ассистента в PACS/RIS

Медицинский продукт создан для врачей, и главная цель ИИ-ассистента — **сократить время описания снимка и снизить утомляемость врача-радиолога**.

### Функционал ИИ-ассистента для DICOM:
1. **Компьютерное зрение (Auto-segmentation / CADe/CADx):** Автоматическая локализация патологий (например, узелков в легких на КТ, новообразований головного мозга на МРТ, очагов пневмонии на рентгене) и отрисовка контуров.
2. **Генеративный ИИ (Auto-Draft Protocol):** Автоматическое составление русскоязычного проекта описания исследования (протокола) на основе распознанных признаков.

### Схема интеграции ИИ:

```
[ Врач во вьювере ] ──( 1. Нажимает "ИИ-Анализ" )──► [ PACS-фасад (FastAPI) ]
         ▲                                                     │
         │ (4. Текст протокола +                               │ (2. Выгрузка серии
         │    координаты патологий)                            │    из Orthanc)
         │                                                     ▼
 [ Экран вьювера ] ◄──( 5. Отрисовка контуров )─── [ ИИ-Сервис (ONNX/VLM) ]
                                                   (Локальный Seg-Model
                                                    или VLM API-коннектор)
```

### Архитектурный контракт API для ИИ:
* `POST /api/v1/studies/{study_uid}/ai-analyze`
  * **Запрос:** `{ "model_type": "lung_nodules" | "brain_mri" | "chest_xray" }`
  * **Процесс:** Бэкенд выгружает DICOM из Orthanc, прогоняет через ONNX-модель (локально) или конвертирует в PNG и отправляет в Vision-Language Model (VLM, например, Gemini-3.5-Flash / Claude-3.5-Sonnet) по API.
  * **Ответ (JSON):**
    ```json
    {
      "status": "success",
      "findings": "В проекции правого легкого (S3) визуализируется округлое образование размером 12x14 мм, контуры четкие. Пневматизация легочных полей сохранена.",
      "impression": "Очаговое образование правого легкого (рекомендуется динамическое наблюдение КТ через 3 мес).",
      "annotations": [
        {
          "slice_index": 45,
          "shape": "circle",
          "coords": { "x": 142, "y": 310, "radius": 15 },
          "label": "Подозрительный очаг 14мм"
        }
      ]
    }
    ```

### Как это экономит время врача:
1. При открытии `ViewerPage.tsx` врач нажимает синюю кнопку **«ИИ-Помощник»**.
2. Вьювер отправляет запрос, получает `annotations` и мгновенно рисует поверх среза №45 (Konva layer в DWV) круглый маркер в месте патологии.
3. Текст из `findings` и `impression` автоматически подставляется как **черновик протокола** в `ProtocolPage.tsx`.
4. Врачу остаётся только проверить разметку, скорректировать текст (экономится до 70% времени "на писанину") и нажать кнопку **«Подписать»**.

## 6. Оптимизация запуска для разработчиков и будущих ИИ-агентов

Для того чтобы исключить путаницу с портами и каталогами, а также дать будущим ИИ-ассистентам возможность запускать всю систему одной командой, разработан корневой оркестратор.

### Единая точка запуска: `start-all-services.bat`
В корневой папке проекта (`C:\Projects\ARCHIVE\MedPlatform\`) создаётся скрипт `start-all-services.bat`. Он автоматически:
1. Проверяет наличие портов (убивает зависшие процессы uvicorn, node, python-static).
2. Поднимает СУБД PostgreSQL в Docker.
3. Запускает Orthanc PACS.
4. Запускает FastAPI бэкенд RIS (:8000) и Очереди (:8005).
5. Запускает Vite-фронтенд (:5173).
6. Запускает NestJS DICOMweb прокси (:3000) и статику вьювера (:5550) для полной интеграции.

Это гарантирует 100% готовность всей платформы "из коробки" без ручной настройки.
