# Ключевые проектные решения

## 1. Next.js App Router вместо Pages Router

**Почему:** App Router — стандарт для новых проектов Next.js. Вложенные layout'ы (например, `/emk/layout.tsx` с сайдбаром) и серверные компоненты по умолчанию.

**Альтернатива:** Pages Router — устаревший, нет вложенных layout'ов без костылей.

**Итог:** App Router + `"use client"` только там, где нужна интерактивность.

## 2. Tailwind CSS v4 (CSS-first) вместо v3

**Почему:** В v4 конфигурация через `@theme` в CSS, а не в `tailwind.config.js`. Это проще, нагляднее, и дизайн-система живёт в одном файле `globals.css`.

**Особенность:** В v4 нет `@apply` по умолчанию (используется редко), нет `@layer`, всё — утилиты.

## 3. Кастомная дизайн-система вместо готовой UI-библиотеки

**Почему:** Проект специфический (медицинская система с казахстанскими полями — ИИН, ОМС). Готовые библиотеки (shadcn/ui, Ant Design) неадекватно масштабируются под узкие отраслевые требования.

**Итог:** Свои компоненты (Badge, CalendarGrid, DoctorCard) на Tailwind-утилитах.

## 4. Route guard через middleware, не на клиенте

**Почему:** Редирект до загрузки страницы, без Flash of Unauthorized Content на клиенте.

**Механизм:** Проверка `auth_token` cookie. Если нет → `NextResponse.redirect('/login?redirect=...')`. Если есть на `/login` или `/register` → редирект на `/dashboard`.

## 5. Мок-данные с API-прослойкой вместо жёсткой вёрстки

**Почему:** Страницы должны быть готовы к подключению реального бэкенда без переписывания UI.

**Решение:** Трёхслойная архитектура:
1. `types/` — интерфейсы
2. `lib/mockData.ts` — мок-данные, аннотированные TODO для бэкенда
3. `lib/api/` — async-функции, использующие мок, но готовые к замене на fetch

## 6. Booking Modal вместо отдельной страницы записи

**Почему:** Пользователь должен видеть список врачей и форму записи в одном контексте, без перехода на другую страницу.

**Механизм:** Модальное окно с шагами: календарь → слоты → форма → подтверждение. Модал управляет своим состоянием (step, selectedSlot), основная страница остаётся нетронутой.

## 7. Фиксированный мобильный хедер с авто-скрытием

**Почему:** На мобилке каждый пиксель экрана ценен. Хедер не должен занимать место при чтении контента, но должен быть доступен при скролле вверх.

**Механизм:** `fixed top-0` + scroll listener: скролл вниз → `-translate-y-full` (скрыт), скролл вверх → `translate-y-0` (показан).

## 8. Валидация форм как чистые функции

**Почему:** Валидация не привязана к React — можно переиспользовать на сервере и в тестах.

**Функции:** `validateLoginForm`, `validateRegisterForm`, `validateAppointmentForm` в `lib/validation.ts`. Каждая возвращает `errors: Record<string, string>`.

## 9. Единый шрифт Inter вместо системного

**Почему:** Inter поддерживает кириллицу, хорошо смотрится в медицинском UI, контрастный и читаемый. Подключен через `next/font/google` с subset `cyrillic`.

## 10. Отказ от CSS-in-JS

**Почему:** Tailwind v4 + CSS-переменные покрывают 100% стилизации. CSS-in-JS (styled-components, emotion) добавляет бандл и сложность без выгоды.

## 11. Цена как строка, а не число

**Почему:** В медицине цена может быть "Бесплатно (ОМС)" или "5000 ₸" с фиксированным форматом. Хранение как строки упрощает отображение без дополнительной форматировки.

**Минус:** Нельзя просто сложить цены. Но в рамках текущих требований (только отображение) это не нужно.

## 12. Структура страниц EMK

layout → sidebar/табы → дочерние страницы. Единый контекст пациента (ФИО, ИИН, группа крови) в сайдбаре, навигация по разделам. Mobile: сайдбар заменяется на горизонтальные табы (overflow-x-auto).
