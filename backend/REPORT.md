## ВВЕДЕНИЕ

Целью данной практической работы является разработка программного комплекса для автоматизации процесса организации и проведения лучевых исследований в медицинском учреждении. Комплекс объединяет три ключевых компонента:

1. **Электронная очередь** — система регистрации пациентов и управления потоками
2. **RIS** (Radiology Information System) — система управления заказами на исследования
3. **PACS** (Picture Archiving and Communication System) — архив медицинских изображений на базе Orthanc

Актуальность работы обусловлена необходимостью цифровизации медицинских учреждений, замены бумажных журналов электронными системами, сокращения времени ожидания пациентов и централизованного хранения медицинских изображений.

## ГЛАВА 1. АНАЛИЗ ПРЕДМЕТНОЙ ОБЛАСТИ

### 1.1. Стандарт DICOM

DICOM (Digital Imaging and Communications in Medicine) — международный стандарт хранения, передачи и обработки медицинских изображений. Ключевые характеристики:

- **Формат файла:** один DICOM-файл содержит графическую информацию (изображение) и метаданные (ФИО пациента, дата, тип исследования, параметры съёмки)
- **Модальности:** CT (компьютерная томография — серия срезов), MR (магнитно-резонансная томография), DX (рентген), US (ультразвук), и др.
- **Сжатие:** Lossy JPEG, JPEG2000, Lossless JPEG
- **Сетевые протоколы:** C-STORE (отправка), C-FIND (поиск), C-MOVE (запрос), C-GET (скачивание)
- **DICOMweb:** RESTful-версия протокола (WADO-RS, QIDO-RS, STOW-RS)

### 1.2. Архитектура PACS-RIS

В реальных медицинских учреждениях используются следующие системы:

- **PACS** — долговременное хранилище DICOM-изображений. Обеспечивает приём от аппаратов (C-STORE), хранение, поиск и выдачу изображений по запросу
- **RIS** — информационная система радиологического отделения. Ведёт заказы, расписание, протоколы. Передаёт метаданные в PACS
- **HIS** (Hospital Information System) — больничная информационная система верхнего уровня

В данной работе реализуется упрощённый аналог RIS с интеграцией через REST API (в реальных системах используется HL7 v2.x).

### 1.3. Обзор существующих решений

| Решение | Тип | Лицензия | Особенности |
|---------|-----|----------|-------------|
| Orthanc | PACS | GPLv3 | Легковесный, REST API, Windows/Linux |
| DWV | Viewer | AGPL | JavaScript, работа в браузере |
| OHIF Viewer | Viewer | MIT | Современный, поддерживает DICOMweb |
| Conquest | PACS | Freeware | Требует настройки DICOM-протоколов |
| DCM4CHEE | PACS | LGPL | Полнофункциональный, сложный в настройке |

Выбор Orthanc обусловлен простотой установки (один exe-файл), наличием REST API и встроенного веб-интерфейса. DWV выбран как лёгкий встраиваемый вьюер.

## ГЛАВА 2. ТЕХНИЧЕСКОЕ ЗАДАНИЕ

### 2.1. Функциональные требования

**Модуль электронной очереди:**
- Регистрация пациента (ФИО, полис, кабинет) с выдачей талона
- Отображение очереди с фильтрацией по кабинету
- Панель врача: вызов следующего, завершение приёма
- Три колонки: на приёме / ожидают / завершённые
- Интеграция с RIS через REST API

**Модуль RIS:**
- Создание заказов на исследования с генерацией StudyInstanceUID
- Хранение заказов в SQLite (журнал)
- Передача метаданных в Orthanc
- Статусы: scheduled → in_progress → completed
- Прокси для Orthanc (обход CORS)

**Модуль просмотра изображений:**
- Загрузка снимков из Orthanc
- Навигация по срезам (колёсико мыши, клавиатура)
- Управление яркостью/контрастом (Window Width/Level)
- Масштабирование и панорамирование
- Пресеты окон для разных модальностей (CT, лёгкие, мозг, кость)
- Отображение метаданных исследования

  │  Эл. очередь      │  │  Панель врача    │  │  Просмотрщик     │
  │  (регистрация)    │  │  (управление)     │  │  (Canvas)        │
  └────────┬─────────┘  └────────┬─────────┘  └────────┬─────────┘
           │ REST                 │ REST                 │ REST (через прокси)
           ▼                      ▼                      ▼
  ┌──────────────────┐  ┌──────────────────┐  ┌──────────────────────┐
  │  СЕРВИС ОЧЕРЕДИ   │  │  СЕРВИС RIS      │  │  PACS (ORTHANC)     │
  │  elqueue/main.py  │  │  ris/main.py     │  │  Порт 8042          │
  │  Порт 8005        │  │  Порт 8000       │  │  /preview, /tools   │
  └──────────────────┘  │  + прокси /orthanc│  └──────────────────────┘
                        └──────────────────┘

### 3.2. Схема базы данных

**RIS (ris.db) — таблица orders:**
- `id` TEXT PK — ID заказа (8 символов)
- `patient_name` TEXT — ФИО пациента
- `patient_id` TEXT — идентификатор (полис)
- `study_uid` TEXT — StudyInstanceUID (DICOM)
- `modality` TEXT — CT / MR / DX / US
- `status` TEXT — scheduled / in_progress / completed
- `created_at` TEXT — дата создания
- `updated_at` TEXT — дата обновления
- `protocol` TEXT — протокол исследования

**Очередь (elqueue.db) — таблица queue:**
- `id` INTEGER PK — автоинкрементный ID
- `patient_name` TEXT — ФИО пациента
- `patient_id` TEXT — ID пациента
- `order_id` TEXT — ID заказа в RIS
- `ticket_number` TEXT UNIQUE — номер талона
- `status` TEXT — waiting / in_progress / done
- `cabinet` TEXT — номер кабинета
- `study_uid` TEXT — StudyInstanceUID
- `created_at` TEXT — дата регистрации

**Сервис очереди (порт 8005):**

| Метод | Путь | Описание |

## ГЛАВА 4. РЕАЛИЗАЦИЯ

### 4.1. Сервис электронной очереди

Реализован в `elqueue/main.py`. Представляет собой FastAPI-приложение с HTML-шаблонами.

**Особенности реализации:**
- Три статуса пациента: waiting → in_progress → done
- Фильтрация по кабинету (101, 102, 103) через параметр `?cabinet=N`
- Интеграция с RIS: при регистрации создаётся заказ через POST /orders
- Генерация талона: 6 случайных буквенно-цифровых символов
- Обработка ошибок: если RIS недоступен, заказ всё равно сохраняется локально

**Интерфейс панели врача (admin.html):**
- Три колонки: "На приёме", "Ожидают", "Завершённые"
- Кнопки: "Вызвать следующего", "Завершить приём", "Просмотр снимков"
- Селектор кабинета для фильтрации

### 4.2. Сервис RIS

Реализован в `ris/main.py`. Обрабатывает заказы, интегрируется с Orthanc.

**Особенности реализации:**
- Генерация корректных DICOM UID (только цифры, без дефисов): `1.2.840.{uuid4_int}`
- Прокси-эндпоинт `/orthanc/{path}` для обхода CORS (Orthanc 1.12.4 не поддерживает CORS-заголовки)
- Прокси передаёт Content-Type от Orthanc к браузеру (image/png, application/json, application/zip)
- Обработка таймаутов при недоступности Orthanc

### 4.3. Просмотрщик DICOM-изображений

Реализован в `ris/templates/viewer.html`. Использует собственный canvas-рендеринг.

**Функциональность:**
- Загрузка срезов через `/orthanc/instances/{id}/preview` (PNG от Orthanc)
- Fallback на `/image-uint8` если `/preview` недоступен
- Навигация по срезам: колёсико мыши, стрелки, слайдер
- Оконное управление (Window Width / Window Level) с пресетами:
  - CT: 400/40
  - Лёгкие: 1500/-500
  - Мозг: 80/40
  - Кость: 350/50
- Автоматический расчёт WW/WL по гистограмме
- Масштабирование (Ctrl+колёсико, кнопки)
- Панорамирование (перетаскивание мышью)
- Отображение координат и значения пикселя
- Боковая панель с метаданными (пациент, исследование, модальность)

### 4.4. Генерация тестовых данных

Реализована в `generate-test-images.py`. Создаёт 4 исследования с реалистичными DICOM-изображениями:

| Исследование | Срезов | Размер | Размер файла | Описание |
|-------------|--------|--------|-------------|---------|
| КТ грудной клетки | 10 | 256×256 | ~131 КБ | Серия аксиальных срезов |
| МРТ поясничного отдела | 8 | 256×256 | ~180 КБ | T2-взвешенные изображения |
| Рентген грудной клетки | 1 | 512×512 | ~525 КБ | Прямая проекция |
| УЗИ брюшной полости | 1 | 256×256 | ~270 КБ | Серошкальное изображение |

Каждое изображение содержит синтезированные пиксельные данные с правдоподобной анатомической структурой (лёгкие, позвоночник, сосуды, печень).

### 4.5. Страница списка исследований

Реализована в `ris/templates/studies.html`. Отображает все исследования в PACS с возможностью:
- Поиска по пациенту, описанию, модальности
- Просмотра (переход во вьюер)
- Скачивания DICOM-архива (ZIP) через `/studies/download/{study_uid}`

### 4.6. Прокси к Orthanc

Ключевое решение: Orthanc 1.12.4 не поддерживает CORS-заголовки `HttpHeaders`. Для работы из браузера (кросс-доменные запросы) все запросы к Orthanc проходят через RIS-прокси:

```
Браузер → /orthanc/preview → RIS → http://localhost:8042/instances/{id}/preview → Orthanc
                                ↓
                          Response(content, media_type=resp.headers.content-type)
```

Прокси поддерживает все HTTP-методы (GET, POST, PUT, DELETE) и корректно передаёт Content-Type.
