# RIS и PACS — что это и зачем

> Простое объяснение для тех, кто видит эти аббревиатуры впервые.
> Документ для практики MedPlatform, 2026.

---

## TL;DR за 30 секунд

**RIS** — это **«кто, когда, что, зачем»** (ресепшен, расписание, врачи, протоколы).
**PACS** — это **«где снимок и как его показать»** (хранилище DICOM + viewer).
Они **всегда разделены** — даже в больших клиниках, где стоят McKesson RIS + Agfa PACS.

В нашей системе:
- **RIS** = FastAPI (`ris/`) + PostgreSQL
- **PACS** = Orthanc (отдельный процесс, C++ сервер)
- **Связь** = REST API (`/api/v1/studies/*` — PACS-фасад в RIS) + DICOM (C-STORE, планируется MWL/MPPS)

## Аналогия из жизни

| Аналогия | RIS-часть | PACS-часть |
|---|---|---|
| **Банк** | Счета клиентов, история операций | Хранилище чеков и сканов |
| **YouTube** | Каналы, подписчики, комментарии | Видеофайлы на серверах |
| **Email** | Список контактов, темы писем | Сами файлы вложений |
| **1С:Документооборот** | Карточки документов, маршруты | Хранилище PDF/Word |
| **Поликлиника** | Ресепшен, очередь, врачи | Архив рентген-снимков + негатоскоп |

> **RIS отвечает на вопросы: КТО? КОГДА? ЗАЧЕМ? КТО ВРАЧ?**
> **PACS отвечает на вопрос: ГДЕ ЛЕЖИТ СНИМОК и КАК ЕГО ПОКАЗАТЬ?**

## Подробно про RIS (Radiology Information System)

**Что делает:**
- Ведёт картотеку пациентов (ФИО, дата рождения, полис)
- Выдаёт талоны в очередь (талон №47, кабинет 101)
- Записывает на исследование (КТ грудной клетки, 14:30)
- Назначает врача (кто смотрит, кто подписывает)
- Хранит протокол (заключение врача после снимка)
- Формирует отчётность (сколько исследований, сколько денег)
- Аудит (кто что сделал и когда)

**Что НЕ делает:**
- Не хранит сами снимки (только ссылки/UID)
- Не показывает снимки (для этого нужен viewer из PACS)

**Аналогия:** это **CRM для рентген-отделения**.

**У нас:** FastAPI `ris/` + PostgreSQL.

---

## Подробно про PACS (Picture Archiving and Communication System)

**Что делает:**
- Принимает DICOM-снимки с аппаратов (КТ, МРТ, рентген, УЗИ)
- Хранит файлы (DICOM — это бинарный формат, .dcm-файлы)
- Архивирует (сжатие: JPEG, JPEG-LS, JPEG2000)
- Ищет по метаданным (QIDO-RS)
- Отдаёт снимки по запросу (WADO, WADO-RS)
- Показывает через viewer (DWV, OHIF, RadiAnt)

**Стандартный набор сервисов PACS:**
- **C-STORE SCP** (порт 4242) — приём DICOM-файлов от аппаратов
- **WADO-URI / WADO-RS** (HTTP) — отдача снимков по UID
- **QIDO-RS** (HTTP) — поиск по метаданным
- **STOW-RS** (HTTP) — приём через HTTP
- **DICOMweb** — общий стандарт для веб-клиентов

**Что НЕ делает:**
- Не знает кто пациент (только DICOM-теги)
- Не знает расписание (нет workflow)
- Не подписывает протоколы

**Аналогия:** это **Dropbox + Photoshop для медицинских снимков**.

**У нас:** Orthanc 26.6.0 (C++ сервер, порты 8042 HTTP + 4242 DICOM).

## Зачем их разделяют (исторически и технически)

### Исторические причины
1. **Разные вендоры.** McKesson делает RIS, Agfa делает PACS, GE делает и то и другое. Клиника покупает лучший RIS + лучший PACS.
2. **Разная цена.** PACS стоит миллионы (лицензии, диски, viewer, техподдержка). RIS — дешевле. Больницы покупают их в разное время.
3. **Разные отделы.** RIS обслуживает IT-отдел и регистратуру. PACS обслуживает радиологи и рентген-лаборанты.

### Технические причины
1. **Разная нагрузка.** PACS — много I/O (большие файлы, терабайты), нужен быстрый диск, RAID. RIS — много SQL (маленькие записи), нужна RAM и индексы.
2. **Разные протоколы.** PACS говорит по DICOM (бинарный, TCP). RIS говорит по HTTP/REST (текст, JSON). Чтобы заменить PACS, не нужно трогать RIS.
3. **Разные требования к надёжности.** PACS нельзя терять снимки (5+ лет хранение, RAID, бэкапы). RIS можно восстановить из бэкапа за час.

### В нашей системе
У нас **небольшая клиника (1 отделение, до 100 исследований/день)** — Orthanc более чем достаточен. Заменять его на dcm4chee (JVM 2-3 ГБ) — overkill. См. TZ.md, ADR-002.

## Как они связаны (в реальной клинике)

```
Ресепшен (RIS)         Аппарат КТ (PACS)         Врач (RIS)
     │                       │                       │
     │  "Пациент Иванов,     │                       │
     │   14:30, КТ"          │                       │
     │                       │                       │
     ├──────► MWL ───────────┤  (аппарат спрашивает: │
     │  (DICOM, 11112)       │   "кого снимать?")    │
     │                       │                       │
     │                       ├─── C-STORE ───────►  │  снимок летит
     │                       │  (DICOM, 4242)        │  в PACS
     │                       │                       │
     │                       │                       │
     │                       ├──── WADO ───────────►│  врач открывает
     │                       │  (HTTP, 8042)         │  снимок
     │                       │                       │
     │  ORM (HL7)            │                       │
     ├─────── заказ ◄────────┼───────────────────────┤
     │                       │                       │
     │                       │   ORU (HL7)           │
     │  протокол ◄───────────┼───────────────────────┤
     │                       │                       │
```

**Стандарты связи:**
- **DICOM MWL** (Modality Worklist) — RIS отдаёт аппарату расписание
- **DICOM MPPS** (Modality Performed Procedure Step) — аппарат сообщает о прогрессе
- **DICOM Storage (C-STORE)** — аппарат шлёт снимки в PACS
- **DICOM WADO** — RIS/PACS отдаёт снимки врачу
- **HL7 ORM/ORU/ADT** — обмен данными с МИС (медицинской информационной системой)
- **IHE XDS/IHE RAD** — профили интеграции (выше, чем просто DICOM/HL7)

**Что видит пользователь:**
- Заходит в `/login`, вводит логин/пароль (RIS)
- Видит очередь в `/` (RIS)
- Регистрирует пациента в `/register` (RIS)
- Врач в `/doctor` вызывает и завершает (RIS)
- Открывает `/studies` (RIS → PACS-фасад → Orthanc)
- Смотрит снимок в `/viewer/:studyUid` (RIS → PACS-фасад → Orthanc preview)
- Пишет протокол в `/protocol/:orderId` (RIS)

**Пользователь не знает про Orthanc.** Всё через RIS.

## Что такое «PACS-фасад в RIS»

**Проблема:** Orthanc предоставляет REST API, но это PACS-API — низкоуровневый:
- `GET /studies` — список ID (нужен ?expand)
- `GET /studies/{id}` — DICOM-теги
- `GET /instances/{id}/preview` — PNG

Чтобы показать пользователю «все снимки с JOIN к пациентам и заказам» — нужно самому делать N запросов и мержить.

**Решение:** PACS-фасад — отдельный слой в RIS, который:
1. Скрывает Orthanc от frontend
2. Объединяет данные (DICOM + RIS order + Patient)
3. Возвращает единый JSON
4. Кэширует превью (TODO)

**Endpoint'ы PACS-фасада** (см. `ris/routers/studies.py`):
```
GET  /api/v1/studies                          — список DICOM-исследований + JOIN
GET  /api/v1/studies/{study_uid}              — детали + серии + инстансы + RIS order
GET  /api/v1/studies/{study_uid}/preview      — превью первого среза (PNG)
GET  /api/v1/instances/{instance_id}/preview  — превью среза (PNG)
GET  /api/v1/instances/{instance_id}/dicom-tags — DICOM-теги
GET  /api/v1/patients/{patient_id}/studies    — снимки пациента
GET  /api/v1/orders/{order_id}/dicom          — DICOM-данные по RIS-заказу
```

**Frontend не знает про `/orthanc/*` прокси.** Всё через `/api/v1/*`.

## DICOM в двух словах

**DICOM** = Digital Imaging and Communications in Medicine.
Стандарт с 1985 года, версия 3.0 с 1993.

**Структура файла:**
```
File Meta Information (128 байт + "DICM")
├── (0002,0000) File Meta Information Group Length
├── (0002,0001) File Meta Information Version
├── (0002,0002) Media Storage SOP Class UID         ← что за снимок
├── (0002,0003) Media Storage SOP Instance UID
└── ...

Dataset (набор тегов)
├── (0008,0060) Modality = "CT"                     ← что за модальность
├── (0010,0010) PatientName = "ИВАНОВ^ИВАН"
├── (0010,0020) PatientID = "001"                   ← ID пациента
├── (0020,000D) StudyInstanceUID = "1.2.840..."     ← UID исследования
├── (0020,000E) SeriesInstanceUID = "1.2.840..."    ← UID серии
├── (0008,1030) StudyDescription = "КТ грудной клетки"
├── (0028,0010) Rows = 512
├── (0028,0011) Columns = 512
├── (7FE0,0010) Pixel Data                          ← сам снимок
```

**Тег** = (group, element) → значение. Например, (0010,0020) = PatientID.

**UID** = уникальный идентификатор в формате `1.2.840.<oid>`. У каждого study, series, instance — свой UID.

**Стандарт кодировки:** Latin-1 (ISO-IR 100) для большинства тегов. Поэтому кириллица в DICOM-тегах выглядит как `???????` — это **нормально** для DICOM, не баг.
