## 1. Цель проекта

Автоматизация работы рентгенологического отделения:
- Электронная очередь пациентов
- Назначение и выполнение исследований (КТ, МРТ, Рентген, УЗИ)
- Подписание протоколов исследований
- Мониторинг загрузки кабинетов и врачей

## 6. Функциональность

### 6.1 Электронная очередь (`/`)
- Отображение очереди по кабинетам
- Фильтрация по кабинету
- Вызов следующего пациента (Call)
- Завершение приёма (Complete)
- Автообновление каждые 10 секунд

### 6.2 Регистрация пациента (`/register`)
- Создание талона с ФИО, полисом, кабинетом, приоритетом
- Автоматическая очередь

### 6.3 Кабинет врача (`/doctor`)
- Выбор кабинета
- Просмотр активных талонов
- Вызов / завершение
- Скелетон загрузки

### 6.4 Заказы RIS (`/orders`)
- Список заказов с фильтрацией по статусу
- Просмотр деталей заказа
- Связь с очередью SmartQ

### 6.5 Пациенты (`/patients`)
- Поиск по ФИО или полису
- Карточка пациента с историей исследований
- Назначение нового исследования

### 6.6 Исследования (`/studies`)
- Просмотр снимков из PACS (Orthanc)
- Связывание снимков с заказами
- DICOM Viewer (DWV)

### 6.7 Протоколы (`/protocol/:orderId`)
- Создание/редактирование протокола исследования
- Черновик и подпись
- После подписи изменения невозможны

### 6.8 Мониторинг (`/monitoring`)
- KPI: заказы сегодня, завершено, в работе, в очереди
- Подписанные протоколы, среднее время обработки
- Загрузка кабинетов
- Активность врачей
- Автообновление каждые 30 секунд

### 6.9 Локализация
- 3 языка: RU, EN, ҚАЗ
- Переключатель в хедере
- Все страницы переведены

## 12. Что сказать руководителю (описание проекта)

### 12.1 Проблематика

В рентгенологических отделениях больниц часто нет единой системы:
- Пациенты записываются в очередь бумажками или «по звонку»
- Врачи не видят, кто следующий
- Нет связи между направлением (заказом) и результатом (протоколом)
- Руководство не может оценить загрузку кабинетов и врачей
- Снимки хранятся отдельно от заключений

### 12.2 Наше решение

Веб-приложение, которое объединяет в себе:

1. **Электронная очередь** — регистратор создаёт талон → пациент встаёт в очередь → врач вызывает → завершает приём. Автообновление каждые 10 секунд.

2. **Назначение исследований** — врач создаёт заказ на КТ/МРТ/Рентген/УЗИ с указанием пациента, модальности, приоритета и клинической информации. Заказ попадает в очередь к рентгенологу.

3. **DICOM-снимки и PACS** — система подключается к Orthanc (PACS-сервер). Врач может просматривать снимки пациента прямо в браузере через DICOM-вьюер с инструментами (измерения, аннотации, контраст).

4. **Протоколы** — по каждому исследованию врач пишет протокол (описание + заключение). Черновик можно сохранять, после подписи изменения блокируются.

5. **Мониторинг** — дашборд с KPI: заказы сегодня/за неделю, среднее время обработки, загрузка кабинетов, активность врачей. Обновляется каждые 30 секунд.

6. **RBAC (ролевая модель)** — 4 роли: администратор, врач, регистратор, техник. Каждая роль видит только свои функции в меню. Все эндпоинты защищены JWT-токенами.

7. **Локализация** — 3 языка (русский, английский, казахский) с переключением без перезагрузки страницы.

8. **Интеграция со SmartQ** — асинхронное подключение к внешнему сервису очереди через REST + Socket.IO для real-time обновлений.

### 12.3 Ключевые архитектурные решения

**Почему React + FastAPI, а не Next.js/Nuxt?**
- Система — SPA без SSR (SEO не нужен — это внутренняя система больницы)
- FastAPI даёт асинхронную обработку запросов с минимальным оверхедам
- Python для backend — стандарт в медицинских системах (простота интеграции с DICOM)

**Почему отдельный сервис очереди (SmartQ)?**
- Очередь — отдельный домен с высокой нагрузкой (много запросов на чтение/запись)
- SmartQ предоставляет встроенный Socket.IO для real-time обновлений
- Можно заменить на любой другой сервис очереди без изменения RIS

**Почему не Axios/React Query?**
- Нативный fetch + кастомная обёртка (client.ts) даёт полный контроль над обработкой ошибок, таймаутами и рефрешем токенов
- React Query добавил бы сложности для простого CRUD-приложения

**Как работает авторизация?**
- JWT access (15 мин) + refresh (7 дней) токены
- Access-токен хранится в памяти (безопасно от XSS)
- Refresh-токен в localStorage для восстановления сессии
- Middleware на всех роутах проверяет роль

### 12.4 «Изюминки» проекта (чем гордиться)

- ✅ **RBAC** с проверкой на каждом эндпоинте (не только на UI)
- ✅ **DICOM Viewer** прямо в браузере (WebAssembly + DWV) с аннотациями
- ✅ **i18n** с 3 языками без перезагрузки
- ✅ **Socket.IO listener** для real-time обновлений очереди
- ✅ **Loading skeletons** на всех страницах вместо пустых таблиц
- ✅ **Таймаут 15с** на все запросы — приложение не виснет при недоступности бэкенда
- ✅ **155+ тестов** (unit + integration) для backend

### 12.5 Возможные вопросы руководителя

| Вопрос | Ответ |
|--------|-------|
| Как обеспечивается безопасность? | JWT-токены с коротким TTL, пароли хешируются bcrypt, проверка роли на каждом эндпоинте, XSS-защита через React |
| Сколько времени разрабатывали? | ~2-3 месяца (итеративно: MVP → RBAC → интеграция → i18n → финальные доработки) |
| Можно ли развернуть в реальной больнице? | Да. Нужен HTTPS, PostgreSQL, backup-стратегия. Frontend собирается в статику, бэкенд — FastAPI под uvicorn/gunicorn |
| Какие метрики производительности? | Страницы грузятся за <1.5с при работающем бэкенде. Мониторинг обновляется каждые 30с. Очередь — каждые 10с |
| Что если SmartQ упадёт? | Таймаут 15с, ошибка показывается пользователю, приложение не виснет. RIS продолжает работать (заказы, пациенты, протоколы) |
| Какая БД используется? | PostgreSQL 16. Миграции через Alembic. Все связи через внешние ключи |

### 13.3 Вариант B: Несколько ноутбуков по сети

> **Требует настройки сети — все ПК в одной локальной сети (Wi-Fi)**

**Настройка:**

1. На серверном ПК узнайте IP-адрес:
   ```bash
   ipconfig
   # → 192.168.1.100 (пример)
   ```

2. Запустите все сервисы на серверном ПК

3. Запустите Vite в режиме разрешённых хостов:
   ```bash
   cd frontend
   npm run dev -- --host 0.0.0.0
   ```

4. На клиентских ПК откройте:
   ```
   http://192.168.1.100:5173
   ```

**Важно:** По умолчанию Vite позволяет подключения только с localhost. Флаг `--host 0.0.0.0` открывает доступ для всех ПК в сети. Для production используйте nginx.

### 13.4 Распределение ролей по ПК для демо

| ПК | Роль в системе | Кто управляет | Действия |
|----|---------------|---------------|----------|
| **Сервер** | Все сервисы | — | Проектор показывает общий экран |
| **ПК #1 (проектор)** | admin / доктор | Докладчик | Основные действия: очередь, заказы, протоколы |
| **ПК #2** | registrar | Помощник | Регистрирует пациентов в реальном времени |
| **ПК #3** | doctor | Помощник | Вызывает/завершает, показывает параллельный приём |

### 13.6 Если что-то пошло не так

| Проблема | Решение |
|----------|---------|
| Не запускается сервис | `netstat -ano \| findstr :PORT` — проверьте, не занят ли порт; завершите процесс: `taskkill /PID <номер> /F` |
| Ошибка CORS на клиенте | Vite Dev Server уже проксирует запросы — CORS не нужен |
| Очередь пуста | Зарегистрируйте нового пациента через `registrar` |
| SmartQ не отвечает | Проверьте лог ошибки. RIS работает и без SmartQ |
| Нет снимков | Orthanc может быть выключен. Пропустите шаг с просмотром снимков |
| Страница не грузится | `Ctrl+Shift+R` (жёсткая перезагрузка без кеша) |
| Кнопка не работает | Проверьте роль пользователя (у техника нет кнопки «Подписать») |
| Забыли пароль | `admin` / `admin123`, `doctor` / `doctor123`, `registrar` / `reg123` |
